# ERP／POS 流程差距與改善順序

## 結論

目前系統已有商品、採購、銷售、庫存、應收應付、傳票、報表、權限及基礎 POS，但還不能宣稱「與鼎新一模一樣」。鼎新各產品、版本與客製流程不同，也需要合法取得的操作規格、畫面與驗收案例，才能逐欄位比對。可行目標應定義為：依客戶實際產業，把單據狀態、權限、庫存與帳務結果做到功能等價，再以使用者驗收測試確認。

目前已修正最危險的流程：送審或核准不再提前入出庫；實際進貨／出貨可分批處理，並在同一資料庫交易內異動庫存、應收應付與傳票；POS 退款必須關聯原交易與原付款；傳票採製單／他人覆核／過帳，已過帳後只能建立反向傳票；月結、年結、重開與關帳後禁止補登均已實作。併發進貨、超量退款、席次搶占及 24 小時離線授權防竄改已有隔離資料庫測試。

## 正確的簡化流程

### 一般企業 ERP

```text
報價 → 銷售訂單草稿 → 送審 → 核准 → 出貨
                                  ├─ 扣實際倉庫庫存
                                  ├─ 建立應收帳款
                                  └─ 建立銷貨傳票

請購（待補）→ 採購單草稿 → 送審 → 核准 → 進貨驗收
                                        ├─ 增加實際倉庫庫存
                                        ├─ 建立應付帳款
                                        └─ 建立進貨傳票

原銷售／採購單 → 退貨申請 → 送審 → 核准 → 退貨過帳
```

### POS 門市

```text
開班 → 掃碼／搜尋商品 → 會員與促銷 → 多元付款 → 結帳
                                                ├─ 原子扣庫
                                                ├─ 建立銷售與收款
                                                ├─ 建立傳票
                                                └─ 收據／發票
退換貨必須讀取原交易 → 驗證可退數量與付款 → 回補庫存 → 退款／折讓
結班 → 現金理論額 → 實點額 → 差異原因 → 日結鎖定
```

## 上線前 P0（不可省略）

1. **部分完成｜資料庫遷移演練**：自動化 migration、升級前後備份、還原、回到舊版結構與全新安裝已通過；正式上線前仍須再用「正式資料匿名化副本」演練。
2. **已完成｜部分進貨／部分出貨**：已有收貨單、出貨單、剩餘量、分批應收應付／傳票、最終尾差與併發超量阻擋。
3. **已完成｜會計關帳**：已有會計期間、月底限制、月結／年結、重開、反向傳票、製單者不可自審及已過帳禁止改刪。
4. **部分完成｜退換貨完整帳務**：POS 已完成原交易、原數量、原成本、原付款比例與庫存／傳票反沖；ERP 銷貨／採購退回仍需正式折讓單、發票折讓及完整傳票。
5. **已完成｜POS 原交易退貨**：已限制原交易、原品項、尚可退數量、原售價／成本、原付款方式與原倉庫，併發超退會被阻擋。
6. **未完成｜台灣電子發票**：串接財政部規格前，不可把一般列印收據宣稱為合法電子發票。
7. **部分完成｜權限分離**：傳票已具製單／送審／他人核准／過帳／沖銷；其他業務單據仍需補互斥角色與金額級距。
8. **部分完成｜備份與災難復原**：還原演練已自動化；每日加密備份、異地保存、保留週期、RTO／RPO 與排程告警仍待部署。

## P1（接近成熟進銷存）

- 請購、採購詢價、供應商比價、採購合約與核准金額級距。
- 報價轉訂單、訂單轉出貨，不重複輸入；保留來源單據與版本。
- 庫存預留、可用量、在途量、批號、序號、效期、儲位、盤點凍結與差異核准。
- 移動加權平均／月加權／標準成本的明確選擇，以及月底成本重算與差異傳票。
- 客戶信用額度、帳齡、催收、付款條件；供應商交期與品質績效。
- 銀行對帳、票據到期、現金流與稅務申報檢核。
- POS 促銷規則、會員點數、價目表、贈品、整單／單品折扣、交班與店長覆核。
- 多門市調撥、總部補貨建議、門市離線交易佇列與衝突處理。

## P2（操作體驗與效能）

- 常用單據採鍵盤優先：Enter、方向鍵、快捷鍵、條碼槍、複製上一筆與批次貼上。
- 首頁改成待辦流程，而不只是圖表：待核准、待進貨、待出貨、逾期應收、低庫存與日結異常。
- 大列表全部伺服器分頁、搜尋與日期區間；不得在管理後台一次載入所有租戶與個資。
- 本機主機可用快取與背景同步，但建立單據、扣庫與編號仍必須在資料庫交易內完成，避免兩台電腦同時超賣。
- AI 只能提出摘要、異常與草稿，不應未經人員確認自動過帳、付款或申報。

## 授權與本機版界線

- 中央端以 Ed25519 私鑰簽發短效租約，本機只有公鑰；更改本機資料不能產生新簽章。
- 試用、租期、方案與席次以中央伺服器時間為準；重裝會占用新裝置席次，不會重置試用。
- Docker 一鍵包作為公司本機主機；Mac／Windows 原生工作站會以 OS 安全儲存區保存 Ed25519 私鑰，逐請求簽章並依 2／3／5／8 台限制。一般瀏覽器沒有工作站私鑰，不能作為付費席次客戶端。
- 防竄改雜湊鏈能「偵測」資料被改，不等於阻止擁有資料庫管理權限的人破壞資料。正式本機版應把每日稽核鏈頭回傳中央端錨定，才能提高事後證明力。
