# 台灣電子發票正式介接準備

## 目前狀態

本系統已完成交易佇列、開立／作廢／折讓事件、失敗重送、相同事件請求碼、防併發字軌取號、完整交易快照及中斷事件復原。`MOCK` 只允許 localhost 本機預覽，產生的 `TEST-` 號碼不可報稅或兌獎。

正式 Turnkey 或 VAN 尚未啟用。沒有測試資格、電子發票專用字軌、憑證、測試帳密、官方 XSD 或選定 VAN 的 API 規格時，系統會明確阻擋正式開立，不能只靠填入環境變數繞過。

## 2026 年介接基準

- 財政部 Turnkey 3.2 僅支援 MIG 4.1 以後 XML 格式。
- 測試平台與正式平台是兩套獨立環境，帳密、字軌及資料不可混用。
- 使用電子發票須取得資格及電子發票專用字軌；新申請者須依所在地國稅局與整合服務平台流程辦理。
- 發票開立後須在規定期限內傳輸；系統以 48 小時作法規時限告警，營運上應盡量即時傳送。

## 正式啟用資料

### Turnkey

1. 營業人名稱與有效統一編號。
2. 測試及正式環境連線帳密。
3. 電子發票專用字軌與取號資料。
4. 有效憑證與 Turnkey 安裝環境。
5. MIG 4.1 官方 XSD、上傳來源目錄、回覆與確認目錄。
6. 開立、作廢、折讓、空白字軌、總分支機構及錯誤重送的驗測結果。

### VAN 加值服務中心

1. VAN 名稱、API 版本及測試／正式端點。
2. Client ID、Client Secret、憑證或 VAN 指定簽章資料。
3. 開立、作廢、折讓、查詢、字軌與載具／捐贈碼驗證規格。
4. 請求冪等碼、逾時後交易查詢及錯誤碼文件。
5. VAN 簽發的測試通過或上線確認文件。

## 上線門檻

1. 測試環境完成一般紙本、手機條碼、自然人憑證、捐贈碼與公司統編情境。
2. 完成開立、作廢、折讓、逾時、重送、回覆遺失及重複請求測試。
3. 驗證繁體中文品名、含稅／未稅、四捨五入、零稅率與免稅資料。
4. 確認正式字軌期間、號碼範圍與賣方統編完全一致。
5. 正式環境先以受控小額交易開立、查詢、作廢／折讓，再由會計與維護人員共同簽收。

未達以上門檻時只能維持 `MOCK` 或「未開立電子發票」，不得把交易收據標示成正式電子發票證明聯。
